## 1단계 — 아키텍처 철학 기초

### 1. Tier vs Layer
> 개발자 필독 | 티어와 레이어 구분 #개발자개념장착 (5분 32초)

| 구분 | Tier | Layer |
|------|------|-------|
| 원래 뜻 | 층, 단계 | 층, 겹 |
| 관점 | **물리적** 분리 | **논리적** 분리 |
| 위치 | 다른 서버 / 프로세스 | 같은 소프트웨어 구성 내 |
| 통신 | 네트워크 호출 | 함수 호출 |
| 예시 | Client ↔ Server ↔ DB | Controller → Service → Repository |

**Tier** : SW System을 구성하는 물리적으로 분리된 실행 계층
- 네트워크로 연결 → 요청/응답이 양방향으로 오감
```
[Client] ↔ [Server] ↔ [DB]
```

**Layer** : SW 구성을 역할과 책임에 따라 논리적으로 나눈 구조
- 함수 호출로 연결 → **의존 방향이 단방향** (상위 → 하위만 호출, 역방향 금지)
```
Presentation (Controller)
      ↓ 함수 호출
  Business (Service)
      ↓ 함수 호출
 Data Access (Repository)
```

> **핵심 차이**: Tier는 ↔ (네트워크 왕복), Layer는 → (의존 단방향)  
> Repository가 Service를 직접 호출하면 Layer 위반 — 의존성 역전이 필요한 경우 DIP 패턴 사용

> 둘은 독립적 — 1-Tier 안에 3-Layer 가능, 3-Tier 각각이 또 내부 Layer 가질 수 있음

**Tier 역사: 2-Tier (Fat Client 시대, 1990년대 ~ 2000년대 초)**

```
[클라이언트 (Fat Client)]   ← 1st Tier
  설치형 데스크톱 .exe
  화면 + 업무로직 + SQL 전부 클라이언트에 존재
       ↕ 직접 연결
      [DB 서버]             ← 2nd Tier
```

- 대표 기술: Visual Basic, Delphi, PowerBuilder, C++ Builder
- 클라이언트가 DB에 직접 접속 권한을 가짐

**2-Tier의 문제점 — 공통 원인: "업무 로직이 클라이언트에 있었기 때문"**

| 문제 | 내용 |
|------|------|
| 보안 | 클라이언트가 DB 직접 접속 권한 보유 → DB 접속 정보 유출 시 직접 접근 가능 |
| 유지보수 | 업무 로직이 여러 클라이언트에 분산 → 정책 변경 시 모든 클라이언트 수정·재배포 필요 |
| 확장성 | 커넥션 풀이 클라이언트 PC마다 분산 → DB가 감당할 커넥션 수를 중앙에서 통제 불가 |

> 이 문제들이 3-Tier (Server 계층 추가) 탄생의 이유

> **핵심**: "통제"는 중간 계층이 있어야 가능. DB는 데이터 저장이 목적이지 트래픽 관리가 목적이 아님.  
> 2-Tier = DB가 최전선. 커넥션 한계 초과 시 그냥 거부, 대기열/안내 없음.  
> 3-Tier = App Server가 트래픽 통제 → DB엔 정해진 수만 연결 (커넥션 풀)

**Tier 역사: 3-Tier (App Server 등장, 1990년대 후반 ~ 2000년대)**

```
[클라이언트 (화면만)]   ← 1st Tier
       ↕ 네트워크
  [Application Server] ← 2nd Tier  ← 업무 로직 여기로 이동
       ↕ 네트워크
      [DB 서버]         ← 3rd Tier
```

App Server의 역할 (2-Tier 문제 전부 해결):
- 업무 로직을 서버로 이동 → 클라이언트 재배포 불필요
- DB 커넥션 풀 중앙 관리 → 확장성 확보
- 트랜잭션 관리
- 클라이언트와 DB 사이의 보안 경계 역할

> C/S 기반: Tuxedo, CORBA, COM+, 초기 EJB  
> Web 기반: WebLogic, WebSphere (WAS) — 웹 시대로 넘어가며 주류

---

**웹/모바일 시대 — 구조 변화 흐름**

| 시기 | 변화 |
|------|------|
| 2000년대 초 | 브라우저가 클라이언트가 됨 → Fat Client 사라지기 시작 |
| RIA 시대 | ActiveX, Flash, Applet으로 브라우저가 다시 무거워짐 → 사실상 Fat Client 재현 |
| **핵심 변화** | **브라우저는 일반적으로 DB에 직접 연결하지 않는다** (DB 드라이버 부재, 보안 정책, 계정 노출 위험) |
| 2010년대 | HTML5 표준화 · ActiveX 퇴출 → 브라우저가 진정한 Thin Client로 / 스마트폰 앱 등장 |
| 현재 | REST API / GraphQL이 프론트-백 경계의 표준 |

> 3-Tier 원칙은 그대로 — 클라이언트 / API 서버 / DB로 이름만 바뀐 것

---

**현대 Architecture — N-Tier**

```
[앱 / 브라우저 클라이언트]
       ↕ HTTPS / REST / GraphQL
      [API 서버]
           ↕
    [캐시 / 메시지큐]
           ↕
      [DB / 저장소]
```

- N-Tier = 3-Tier 이상으로 계층이 늘어난 구조
- 캐시, 메시지큐, 인증 서버 등이 별도 Tier로 분리
- MSA에서는 여러 개의 3-Tier 서비스가 공존

> Tier가 늘수록 유연성↑ 복잡성↑ — 적절한 수준이 중요

**Tier "물리적 분리"의 현대적 해석**
- 고전: 기계(서버)가 물리적으로 다름
- 현대 기준: **네트워크 없이는 통신 불가능한 경계**가 존재하면 Tier로 취급
- Docker 컨테이너도 Tier — 같은 기기여도 HTTP/TCP 쳐야 통신 가능
- 판단 기준: `네트워크 호출 필요?` → YES = Tier, `함수 호출로 끝?` → YES = Layer

```
실제 사례 (Spring + MyBatis 환경):
[브라우저] ↔ [Spring 서버] ↔ [MySQL]   ← Tier (네트워크 경계)
              ↑ 서버 내부
         Controller (@Controller)
              ↓
          Service (@Service / ServiceImpl)
              ↓
         Repository (@Mapper / MyBatis)    ← Layer (함수 호출)
```

**Layer 단방향이어야 하는 이유**
- 역방향 호출 시 **순환 의존** 발생 → 수정 한 곳이 전체에 영향
- 계층 간 **책임 경계 붕괴** → 어디서 뭘 해야 하는지 모호
- **테스트 불가** → Repository 단위 테스트하려면 Service, Controller까지 올려야 함
- `Service` 인터페이스 + `ServiceImpl` 구현체 분리 = 상위 계층이 구현체가 아닌 **추상에만 의존**하도록 강제 (각 계층은 자기 바로 아래 인터페이스만 알면 됨)

**[심화] 웹 vs 게임 통신 방식 비교**

| | 웹 | 실시간 게임 |
|--|--|--|
| 연결 | 요청마다 새로 (HTTP) | 상시 유지 (TCP Socket / WebSocket) |
| 상태 유지 | 세션ID 쿠키 | 연결 자체가 상태 |
| 프로토콜 | TCP | 위치/이펙트 → UDP, 결제/채팅 → TCP |

**HTTP는 Stateless** — 요청/응답 후 연결 끊김. 로그인 유지는 세션ID 쿠키로 흉내냄.

**게임 데이터 분리:**
```
서버 보유 (검증 필요): 위치, 체력, 골드, 스킬 판정
클라 보유 (렌더링용): 스킨, 모델, 이펙트, 미니맵
→ 커스텀 스킨이 본인 화면에만 적용되는 이유
```

**UDP 쓰는 이유** — 위치 패킷 하나 손실돼도 다음 프레임에 새 위치 옴. 재전송 기다리는 딜레이가 더 치명적.  
**보간(interpolation)** 으로 손실된 위치 추정. 스킬 적중 판정은 서버가 최종 결정.  
→ "분명히 피했는데 맞음" = 핑 차이 + UDP 손실 + 서버 판정 타이밍 차이

**[심화] 브라우저 제약**

브라우저 = 샌드박스. TCP/UDP 직접 못 씀. 실시간 연결 우회책만 허용:
- `WebSocket` — HTTP 업그레이드 후 양방향 유지 (채팅, 주식차트)
- `SSE` — 서버→클라 단방향
- `WebRTC` — 브라우저끼리 P2P (화상통화)

게임 클라이언트(.exe) = OS 위 직접 실행 → TCP/UDP 자유, 제약 없음.  
브라우저 게임 성능 한계 / 플래시 망한 이유 = 이 샌드박스 제약 때문.

### 2. 모놀리식이 먼저입니다
> 모놀리식이 먼저 입니다. 😊 (4분 3초)

**Why MSA?**

MSA는 모놀리식 아키텍처의 한계에서 출발한 해결책.  
그 한계를 모르면 MSA도 이해할 수 없음.

> "모놀리식을 제대로 해본 사람이 MSA를 제대로 씁니다"

**모놀리식이 커지면 나타나는 대표적인 한계**  
(Martin Fowler & James Lewis, 2014 / Conway's Law 기반)

| 한계 | 문제 | MSA 해결책 |
|------|------|-----------|
| 배포 | 버튼 하나 바꿔도 전체 애플리케이션 재배포. 배포 범위·위험도 증가 | 서비스 단위 독립 배포 |
| 확장 | 특정 기능만 과부하여도 전체 애플리케이션을 함께 확장 | 서비스 단위 독립 스케일 |
| 협업 | 팀이 커질수록 변경 하나가 다른 팀 작업을 망침. 독립 개발 어려움 | 팀 단위 독립 개발 지원 |

> 이 한계를 직접 경험해봐야 MSA의 설계 결정들이 납득됨

**[추가] 모놀리식 한계 보완**

| 한계 | 문제 | MSA 해결책 |
|------|------|-----------|
| 장애 격리 (SPOF) | 결제 모듈 하나 죽으면 전체 서비스 다운 | 서비스별 독립 장애 격리 |
| 스케일링 비효율 | 트래픽 몰리는 기능만 키우고 싶어도 전체 애플리케이션을 스케일업해야 함 | 트래픽 몰리는 서비스만 스케일아웃 (주문 서비스만 10개 띄우기) |
| 기술 스택 고착 | 전체가 한 언어/프레임워크에 묶임 | 서비스별 최적 스택 선택 가능 (추천 → Python, 결제 → Java) |

**[추가] MSA 단점 상세**

영상에서 "관리 복잡성" 으로 묶인 것의 실체:

| 단점 | 내용 |
|------|------|
| 분산 트랜잭션 | 주문 + 결제 + 재고 동시 성공해야 하는데 중간 하나 실패 시 롤백 처리 매우 복잡 |
| 네트워크 레이턴시 | 함수 호출이 네트워크 호출로 바뀜 → 느려짐, 실패 가능성 추가 |
| 데이터 일관성 | 서비스마다 DB 분리 → 정합성 맞추기 어려움 |
| 배포 파이프라인 | 서비스 수만큼 CI/CD 파이프라인 관리 |
| 로컬 개발 | 서비스 여러 개 동시 기동해야 개발 가능 |

> 결론: 모놀리식 문제 = **결합도**, MSA 문제 = **분산 복잡성**. 트레이드오프.  
> 시스템이 충분히 크지 않으면 MSA가 오히려 일을 늘림.

### 3. 모놀리스 vs MSA
> 스타트업 딜레마! 모놀리스냐 MSA냐? (7분 20초)

| 구분 | 모놀리스 | MSA |
|------|----------|-----|
| 배포 | 단순 | 복잡 |
| 확장성 | 비효율적 | 효율적 |
| 장애 격리 | 취약 | 우수 |
| 기술 복잡도 | 낮음 (구조) | 높음 (운영) |

**콘웨이의 법칙 (Conway's Law)**
> 시스템의 구조는 그것을 만드는 조직의 커뮤니케이션 구조를 닮아간다는 법칙.

**[3번 영상 복습 질문]**
1. 스타트업에서 모놀리스와 MSA 중 무엇을 선택해야 하는가?

    힌트 각도:
    - 사업 불확실성 — 초기에 규모를 알 수 있는가?
    - 트래픽 — 팀 크기보다 트래픽 폭발 가능성이 더 중요할 수도
    - 팀 구조 — Conway's Law: 조직이 분리 안 됐으면 MSA 해도 모놀리식처럼 됨
    - 전환 전략 — 모놀리식 먼저 → 한계 느낄 때 분리 (Martin Fowler 권장)
    - 인프라 비용 — MSA는 Docker/K8s/Gateway 등 초반 셋업 비용 큼

2. MSA에서 서비스 간 트랜잭션은 어떻게 처리해야 하는가?
3. 수많은 MSA 서비스들은 어떤 방식으로 통신하는 것이 좋은가?

### 4. 모듈러 모놀리스

> 모놀리스 ≠ 구조 없음. 그래도 시스템이 커지면 **덩어리가 너무 커지는 문제**는 남음.

---

**MSA의 등장**

> "언젠가부터 너도나도 MSA로… MSA = 현대적인 아키텍처"라는 분위기

- MSA의 약속: 서비스별 독립 배포, 개별 확장, 장애 격리
- 대형 사례(Netflix·Amazon): 수백 개 마이크로서비스, 수천 명 개발자 조직, 엄청난 트래픽 규모

**그러나 부작용이 서서히 드러났다**

| MSA의 현실 (운영 복잡도 급증) | 팀이 작으면 역효과 |
|-------------------------------|-------------------|
| 네트워크 통신 / 장애 포인트 증가 | 서비스 수 > 감당 가능한 팀 수 |
| 분산 트랜잭션 처리 | Conway's Law — 조직이 아키텍처를 결정 |
| 모니터링·배포 파이프라인 부담 | 복잡성이 오히려 속도를 늦춤 |

> "우리 팀 규모에 MSA가 맞나?"

**모듈러 모놀리스**

> 그냥 모놀리스 말고, **모듈러 모놀리스**  
> 단일 배포 단위를 유지하면서, 내부를 도메인 경계로 명확히 분리한 구조

**세 가지 아키텍처 비교**

| 구분 | 모놀리스 | 모듈러 모놀리스 | MSA |
|------|----------|----------------|-----|
| 경계 | 없음 | 도메인 경계 강제 | 독립 배포 단위 |
| 배포 | 단일 | 단일 | 서비스별 독립 |
| 운영 복잡도 | 낮음 | 낮음 | 높음 |
| 특징 | 유지보수 어려움 | 도메인 분리 유지 | 개별 확장 가능 |

**언제 무엇을 선택하는가?** (절대적 기준 없음)

| 기준 | 모듈러 모놀리스 | MSA |
|------|----------------|-----|
| 팀 규모 | ~10명 이하 | 팀이 서비스별로 분리될 수 있을 때 |
| 배포 독립성 | 불필요 or 낮음 | 서비스별 독립 배포 필수 |
| 도메인 복잡도 | 중간 수준 | 높고, 도메인별 성장 속도가 다를 때 |

**전환 전략 — 모놀리스 → 모듈러 → MSA**

> 모듈러 모놀리스는 MSA의 예행연습. 경계를 코드로 강제해두면 필요할 때 쪼개기가 훨씬 수월함.

```
1단계: 모놀리스 정리     → 도메인 경계 식별, 의존성 정리
2단계: 모듈러 모놀리스   → 경계를 코드로 강제, 인터페이스 명확화

```

> "모듈러 모놀리스는 MSA 가기 전 임시방편 아닌가요?"  
> → 아님. 서비스 규모와 팀 상황에 따라 **최종 목적지**가 될 수 있음. MSA는 필연적 목표가 아님.

## 2단계 — DB / 트랜잭션

### 5. 트랜잭션 ACID
> 백엔드 개발자라면 반드시 알아야 하는, 트랜잭션 ACID (5분 54초)

**트랜잭션이란?**
> 여러 DB 연산을 하나의 논리적 작업 단위로 묶은 것. 전부 성공하거나 전부 없던 일이 되어야 함.

```
예: 계좌이체
1. A 계좌 -10,000원
2. B 계좌 +10,000원
→ 1만 성공하고 2가 실패하면? DB는 절대 그 상태로 남으면 안 됨
```
---

**A — Atomicity (원자성)**

> 트랜잭션 내 모든 연산은 전부 성공(COMMIT) 또는 전부 실패(ROLLBACK). 중간 상태 없음.

- "All or Nothing"
- 실패 시 DB는 트랜잭션 시작 전 상태로 되돌아감
- 구현: UNDO 로그 (실패 시 이전 값으로 복구)

---

**C — Consistency (일관성)**

> 트랜잭션 전후로 DB의 무결성 제약 조건이 항상 유지되어야 함.

- 잔액이 음수가 되면 안 된다는 규칙 → 트랜잭션 후에도 유지
- NOT NULL, UNIQUE, FK 제약 등 모두 포함
- 비즈니스 규칙을 DB가 보장

```
잔액 >= 0 제약이 있다면:
A 계좌 잔액 5,000원인데 10,000원 이체 시도 → 트랜잭션 거부
```

---

**I — Isolation (격리성)**

> 동시에 실행 중인 트랜잭션들은 서로의 중간 상태를 볼 수 없어야 함.

- 트랜잭션 A가 진행 중일 때 트랜잭션 B는 A의 미완료 변경사항을 못 봄
- **완벽한 격리 = 성능 저하** → 격리 수준(Isolation Level)으로 트레이드오프 조절
- 다음 영상(6번)에서 격리 수준 4단계 상세 다룸

| 격리 낮음 ↔ 높음 | 문제 현상 |
|--|--|
| 격리 낮으면 | Dirty Read, Non-Repeatable Read, Phantom Read 발생 |
| 격리 높으면 | 동시성 낮아짐, 성능 저하 |

---

**D — Durability (지속성)**

> 커밋된 트랜잭션의 결과는 장애가 발생해도 영구적으로 유지되어야 함.

- DB 서버 다운, 정전 → 재시작 후에도 커밋된 데이터 보존
- 구현: REDO 로그 (커밋 후 장애 시 로그로 복구)
- `COMMIT` 응답을 받았다면 디스크에 쓰여진 것이 보장됨

---

**ACID 한눈에**

| 속성 | 핵심 한 줄 | 구현 수단 |
|------|-----------|----------|
| Atomicity | 전부 or 없음 | UNDO 로그, ROLLBACK |
| Consistency | 제약 조건 항상 유지 | DB 제약, 비즈니스 규칙 |
| Isolation | 중간 상태 안 보임 | 락, MVCC |
| Durability | 커밋 = 영구 보존 | REDO 로그, WAL |

> ACID는 DB가 "믿을 수 있는 저장소"임을 보장하는 4가지 약속.  
> 이 중 하나라도 깨지면 데이터 정합성 보장 불가.

**Spring에서 트랜잭션**

```java
@Transactional  // 이 어노테이션 하나가 ACID 전부 보장
public void transfer(Long fromId, Long toId, int amount) {
    accountRepository.decrease(fromId, amount);  // 실패하면
    accountRepository.increase(toId, amount);    // 전체 ROLLBACK
}
```

2. Consistency는 누가 보장하는가 — DB인가 애플리케이션인가?
3. Isolation이 완벽하면 왜 성능이 낮아지는가?

### 6. 트랜잭션 격리수준
> 트랜잭션 격리수준 4단계 완전 이해 (12분 19초)

**핵심 면접 질문**
> "왜 무조건 최고 수준(SERIALIZABLE)으로 설정하지 않나요?"  
> → 격리 높을수록 락 범위 넓어짐 → 대기 트랜잭션 쌓임 → 처리량 저하. 신뢰성과 성능의 트레이드오프.

---

**격리가 낮을 때 생기는 문제 현상 3가지**

| 현상 | 설명 | 예시 |
|------|------|------|
| Dirty Read | 커밋 안 된 값 읽음 | A가 롤백할 값을 B가 이미 읽어서 사용 |
| Non-Repeatable Read | 같은 쿼리 두 번 날렸는데 결과 다름 | 트랜잭션 중 다른 트랜잭션이 UPDATE·DELETE |
| Phantom Read | 같은 조건 두 번 조회했는데 행 수 다름 | 트랜잭션 중 다른 트랜잭션이 INSERT |

```
Dirty Read:
  A: UPDATE money = 0           (미커밋)
  B: SELECT money → 0 읽음
  A: ROLLBACK → 다시 10000
  B는 존재한 적 없는 0을 읽은 것

Non-Repeatable Read:
  B: SELECT money → 10000
  A: UPDATE money = 5000; COMMIT
  B: SELECT money → 5000  ← 같은 트랜잭션인데 다른 결과

Phantom Read:
  B: SELECT COUNT(*) WHERE age > 20 → 5건
  A: INSERT 새 행 (age=25); COMMIT
  B: SELECT COUNT(*) WHERE age > 20 → 6건  ← 유령 행 등장
```

---

**격리 수준 4단계 (낮음 → 높음)**

| 단계 | 수준 | Dirty Read | Non-Repeatable Read | Phantom Read | 성능 |
|------|------|-----------|---------------------|--------------|------|
| 1단계 | READ UNCOMMITTED | 발생 | 발생 | 발생 | 최고 |
| 2단계 | READ COMMITTED | 방지 | 발생 | 발생 | 높음 |
| 3단계 | REPEATABLE READ | 방지 | 방지 | 발생 | 중간 |
| 4단계 | SERIALIZABLE | 방지 | 방지 | 방지 | 최저 |

---

**각 수준 설명**

**READ UNCOMMITTED** — 커밋 안 된 것도 다 읽음. 실무 사용 거의 없음. Dirty Read 허용.

**READ COMMITTED** — 커밋된 것만 읽음. **PostgreSQL·Oracle 기본값**. 대부분 서비스에서 충분.

**REPEATABLE READ** — 트랜잭션 시작 시점 스냅샷 고정. 같은 SELECT 항상 같은 결과. **MySQL InnoDB 기본값**. MVCC로 구현.

**SERIALIZABLE** — 완전 직렬 실행. 모든 문제 방지. 락 경합 심해 처리량 급락. 금융 일부 배치 처리에서만 사용.

---

**MVCC (Multi-Version Concurrency Control)**

> MySQL InnoDB가 REPEATABLE READ를 성능 저하 없이 구현하는 방법

락 없이 격리 구현:
- 트랜잭션 시작 시점에 스냅샷 찍음 (Undo 로그에 이전 버전 보관)
- 트랜잭션 안에서 SELECT는 스냅샷 기준으로 읽음
- **남이 바꾼 건 스냅샷 기준으로 무시, 내가 직접 바꾼 건 즉시 반영**
- 읽기와 쓰기가 서로 블로킹 안 함

```
B 트랜잭션 시작 → 스냅샷: money=10000

A: UPDATE money=5000; COMMIT   ← DB 실제값 바뀜
B: SELECT money → 10000        ← 남이 바꾼 건 스냅샷 기준

B: UPDATE money=7000           ← 본인이 직접 바꿈
B: SELECT money → 7000         ← 본인 변경은 즉시 반영

B 트랜잭션 종료 후 새 조회 → 7000 (최신값)
```

---

**실무 기본값**

| DB | 기본 격리 수준 |
|----|--------------|
| MySQL InnoDB | REPEATABLE READ |
| PostgreSQL | READ COMMITTED |
| Oracle | READ COMMITTED |
| SQL Server | READ COMMITTED |

> Spring `@Transactional`은 DB 기본값 따라감. 변경 필요 시 `@Transactional(isolation = Isolation.SERIALIZABLE)`

---

**SELECT ... FOR UPDATE (비관적 락)**

> 스냅샷 무시하고 실제 최신값 읽으면서 동시에 락 걸기. "읽고 → 판단 → 쓰는" 패턴에서 사용.

```sql
SELECT * FROM item WHERE id = 1 FOR UPDATE;
-- 해당 행에 쓰기 락(exclusive lock)
-- 다른 트랜잭션의 UPDATE/DELETE → 대기
-- 다른 트랜잭션의 FOR UPDATE SELECT → 대기
-- 다른 트랜잭션의 일반 SELECT → 통과 (스냅샷 읽기라 락 안 봄)
```

스냅샷 읽기는 "남이 바꾼 거 안 보임" → 재고 10 읽고 차감했는데 실제론 이미 0이면 문제.  
`FOR UPDATE`는 최신값 읽고 **수정만 막음** (일반 읽기는 허용). 내 트랜잭션 끝날 때까지 다른 트랜잭션 수정 대기.

```java
// Spring Data JPA
@Lock(LockModeType.PESSIMISTIC_WRITE)  // FOR UPDATE
Item findByIdForUpdate(@Param("id") Long id);

// 사용
@Transactional
public void order(Long itemId) {
    Item item = itemRepo.findByIdForUpdate(itemId);  // 최신값 + 락
    if (item.getStock() < 1) throw new SoldOutException();
    item.decreaseStock();  // 안전하게 차감
}
```

---

**Gap Lock (MySQL InnoDB 전용)**

> 행 자체가 아니라 **행 사이의 빈 공간**에 락 거는 것. Phantom Read 방지 목적.

```
현재 age 데이터: 10, 20, 30

B: SELECT * WHERE age > 20
   → (20초과~30 사이, 30 이후 공간에 Gap Lock 걸림)

A: INSERT age=25  ← 그 공간에 삽입 시도 → B 끝날 때까지 대기
A: INSERT age=35  ← 마찬가지 대기
```

REPEATABLE READ는 ANSI 표준상 Phantom Read 허용인데, InnoDB가 Gap Lock으로 추가 방어.  
→ MySQL에서 3단계(REPEATABLE READ)로도 Phantom Read 거의 안 터지는 이유.

PostgreSQL은 Gap Lock 없고 MVCC를 더 정교하게 구현해서 다른 방식으로 처리.
